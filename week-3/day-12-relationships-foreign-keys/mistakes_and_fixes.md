# Mistakes and Fixes - Day 12

These are the real errors I got when I tried to break the database on purpose.
The point is not to avoid mistakes. The point is to understand why the error
happened and what the correct insert should look like.

---

## Foreign Key Test 1 - Bad customer_id

**What I tried:**
`sql
INSERT INTO orders (customer_id, order_date, status, channel)
VALUES (9999, '2024-01-15', 'Pending', 'Online');
`

**Error received:**
`FOREIGN KEY constraint failed`

**Why it happened:**
customer_id 9999 does not exist in the customers table. The orders table has a
FOREIGN KEY on customer_id that points to customers(customer_id). SQLite checked
the customers table, found nothing with that ID, and blocked the insert.

Also, I forgot to run PRAGMA foreign_keys = ON at the start. Without that line,
SQLite ignores foreign key rules completely. That was my first surprise.

**Correct insert:**
`sql
-- First make sure the customer exists
INSERT INTO customers (customer_name, city, segment)
VALUES ('Ensar Aliu', 'Prizren', 'Individual');

-- Now use the real customer_id (1, generated by AUTOINCREMENT)
INSERT INTO orders (customer_id, order_date, status, channel)
VALUES (1, '2024-01-15', 'Pending', 'Online');
`

## Foreign Key Test 2 - Bad order_id in order_items

**What I tried:**
`sql
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (9999, 1, 2);
`

**Error received:**
`FOREIGN KEY constraint failed`

**Why it happened:**
order_id 9999 does not exist in the orders table. The order_items table requires
every order_id to point to a real order. There was nothing to point to.

**Correct insert:**
`sql
-- order_id 1 and product_id 1 must already exist
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (1, 1, 2);
`

## Foreign Key Test 3 - Bad product_id in order_items

**What I tried:**
`sql
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (1, 9999, 1);
`

**Error received:**
`FOREIGN KEY constraint failed`

**Why it happened:**
product_id 9999 does not exist in the products table. Same logic as before.
The foreign key checked the products table and found nothing matching that ID.

**Correct insert:**
`sql
-- Insert a real product first
INSERT INTO products (product_name, category, price)
VALUES ('Wireless Mouse', 'Accessories', 25);

-- Now use product_id 1
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (1, 1, 1);
`

## CHECK Test 1 - Price of 0 and negative price

**What I tried:**
`sql
INSERT INTO products (product_name, category, price)
VALUES ('Broken Item', 'Electronics', 0);

INSERT INTO products (product_name, category, price)
VALUES ('Negative Price Item', 'Office', -50);
`

**Error received:**
`CHECK constraint failed: products`

**Why it happened:**
The schema has CHECK (price > 0) on the products table. Zero is not greater than
zero, and -50 is definitely not. Both got blocked immediately.

**Correct insert:**
`sql
INSERT INTO products (product_name, category, price)
VALUES ('Wireless Mouse', 'Accessories', 25);
`

## CHECK Test 2 - Quantity of 0

**What I tried:**
`sql
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (1, 1, 0);
`

**Error received:**
`CHECK constraint failed: order_items`

**Why it happened:**
The schema has CHECK (quantity > 0) on order_items. A quantity of zero means
nothing was ordered, which makes no sense as a line item.

**Correct insert:**
`sql
INSERT INTO order_items (order_id, product_id, quantity)
VALUES (1, 1, 2);
`

## Status Test - Using 'done' instead of 'Completed'

**What I tried:**
`sql
INSERT INTO orders (customer_id, order_date, status, channel)
VALUES (1, '2024-01-15', 'done', 'Online');
`

**Error received:**
`CHECK constraint failed: orders`

**Why it happened:**
The schema uses CHECK (status IN ('Pending', 'Cancelled', 'Completed')). The
value 'done' is not in that list. SQLite is case-sensitive and exact, so even
'completed' with a lowercase c would fail. It has to match exactly.

**Correct insert:**
`sql
INSERT INTO orders (customer_id, order_date, status, channel)
VALUES (1, '2024-01-15', 'Completed', 'Online');
`

## What I Learned From All of This

The database is not just storing data. It is enforcing rules. Every mistake
above would have silently passed in a table with no constraints, and I would
have had dirty data sitting in the system without knowing it.

Foreign keys keep relationships real. CHECK constraints keep values meaningful.
Without both, data quality is entirely on the developer to manage manually,
which is way harder and way more error-prone.
